## 🔴 ACTIVE BUG — "data immediately getting deleted" on the phone

**Your data is SAFE. This is NOT real data loss.** Evidence gathered this session:
- `~/DataForge` git history has **zero note deletions** (only the sqlite `-wal`/`-shm`
  sidecars were ever removed, by a gitignore chore). 8 note files on disk.
- The index has **0 rows with `deleted=1`**.

### Confirmed root cause: server/client VERSION MISMATCH
- The **running launchd server on :5040 is OLD code (pre-M1)**. Proof:
  - `curl "http://localhost:5040/api/changes?since=0"` returns docs with **no
    `archived` field**.
  - `~/DataForge/meta/index.sqlite` has **no `archived` column** (`sqlite3` →
    `no such column: archived`).
- The **phone runs the NEW M6 web client**, which expects `archived`.
- `make deploy` rebuilt the web (new UI shipped — "flows are better") but the
  **server process was never restarted onto the new code**, so its boot
  migration never ran.

### The exact "disappear" mechanism — CONFIRMED + client fix APPLIED
`packages/core/src/views.ts:49` (`matchesView`) did:
`if ((f.archived ?? false) !== doc.archived) return false;`
The old server omits `archived`, so `doc.archived` is `undefined`, and
`false !== undefined` is **true** → EVERY old-server note is filtered out of
EVERY normal view. That is the "data getting deleted" (it's hidden, not deleted).

**FIXED client-side this session** (committed, unpushed): coalesced the doc side
too — `(f.archived ?? false) !== (doc.archived ?? false)`. The client now
tolerates a missing `archived` field, so notes show even against the old server.
Typecheck + 43 core tests green. **Still redeploy the server** (below) to fully
match schemas; and hard-refresh the phone PWA so it picks up this new client.
- **TODO:** add a regression test in `packages/core/test/views.test.ts` — a doc
  with `archived: undefined` must match the `all` view.
- Could NOT reproduce deletion in a clean browser (capture/edit/close stable, no
  server-side deletes in the log) — because my clean browser talked to a NEW
  server that sends `archived`; the phone hit the OLD one. That's the whole bug.
  The only client *delete* path is the editor trash button + `window.confirm`
  (ui.tsx:596) — a separate accidental-tap concern (see mobile polish).

### THE FIX (do this first next session)
1. **Redeploy the server** so it runs current code + migrates:
   `make deploy` — rebuilds web AND `launchctl kickstart -k com.alcatraz.forge-server`.
   Boot runs `openDb → migrateColumns` = `ALTER TABLE docs ADD COLUMN archived
   INTEGER NOT NULL DEFAULT 0` (additive, tested M1, safe — 8 rows get archived=0).
2. **Verify:** `curl "http://localhost:5040/api/changes?since=0"` now shows
   `"archived":false` in each doc; `sqlite3 ~/DataForge/meta/index.sqlite
   "PRAGMA table_info(docs)"` shows the `archived` column.
3. **On the phone:** hard-refresh / reinstall the PWA to clear any stale
   service-worker bundle + Dexie, then retest capture/edit.
4. If deletion PERSISTS after 1–3, harden the client to tolerate a missing
   `archived` field (defensive `!doc.archived`), and re-check the trash-button
   theory below.
